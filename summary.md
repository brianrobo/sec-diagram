# FBS 평판 관리 시스템 - Threat Scenario & Recovery Flow

## 1. 시나리오 개요

**시스템 목표**: 공격 시나리오 기반 동적 평판 점수 관리 및 자가 복구 메커니즘

**핵심 메커니즘**:
1. **Threat Scenario Detection**: 사전 정의된 공격 워크플로우 패턴 매칭
2. **Dynamic Scoring**: 위협 감지 시 Untrust Score 증가
3. **Trust Server Reporting**: 전체 네트워크에 위협 정보 공유
4. **Self-Recovery**: AKA 성공 시 점수 감소로 False Positive 해소   // (me) false positive(오탐) => 정상 기지국인데, 공격 기지국으로 잘못 판단하여 차단해 버리는 상황.
5. **Extensible Architecture**: 새로운 공격 시나리오 추가 용이

**특허 핵심**: 워크플로우 기반 패턴 매칭 + 협업 방어 + 자가 복구



2. Self-Recovery가 False Positive를 해결하는 방식
질문자님의 아이디어에서 AKA(인증) 성공 시 점수를 낮추는 로직이 왜 오탐을 줄이는지 그 메커니즘은 다음과 같습니다.

상황 설정: 공격자가 실제 운영 중인 A 기지국의 ID를 도용해 공격을 했습니다. 서버는 A 기지국의 점수를 높여 '차단' 상태로 만듭니다.

기존 방식의 문제: 공격자가 사라진 뒤, 진짜 A 기지국이 나타나도 서버에는 여전히 '공격 기지국'으로 등록되어 있어 사용자들이 접속을 못 합니다. (이것이 지속되는 오탐 상태입니다.)

본 발명의 해결책:

사용자가 진짜 A 기지국에 접속을 시도합니다.

진짜 기지국은 핵심망(Core Network)과 연결되어 있으므로 AKA(인증) 절차를 완벽히 수행할 수 있습니다. (가짜 기지국은 AKA를 절대 통과시킬 수 없습니다.)

AKA 성공 리포트가 서버에 전달되는 순간, 서버는 "아, 이 Cell ID를 쓰는 놈은 이제 진짜(정상)구나!"라고 판단합니다.

**가변 가중치(EMA)**에 의해 점수가 즉시 깎이면서 차단이 해제됩니다.

즉, "잘못된 판단(False Positive)을 내렸더라도, 정상적인 절차(AKA)가 확인되는 즉시 그 판단을 스스로 수정(Recovery)하여 오탐 시간을 최소화한다"는 뜻입니다.



3. 특허 관점에서의 기술적 가치
이 로직이 포함되면 특허 검토 시 다음과 같은 강력한 논거가 됩니다.

가용성 보장 (Availability): 보안 강화를 위해 차단 임계치를 낮게 설정하더라도, 복구 로직이 확실하기 때문에 "과감한 차단"이 가능해집니다.

지능적 변별력: 단순히 "공격이 있었다"는 과거의 사실에만 얽매이지 않고, **"현재 접속 시도가 정당한가"**를 실시간으로 반영하는 동적 시스템임을 증명합니다.


요약하자면
**"실제 기지국 ID를 도용당해 억울하게 블랙리스트에 올라갔던 정상 기지국이, AKA라는 확실한 증거를 통해 자신의 결백을 증명하고 다시 서비스 대열에 합류하는 과정"**을 의미합니다.

이 부분을 프리뷰 미팅에서 강조하시면, "공격자가 ID 도용해서 정상 망 마비시키면 어떡할 거냐?"라는 예상 질문에 완벽한 방어 답변이 됩니다.
